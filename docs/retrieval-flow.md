# Retrieval flow
# aiLog — 자연어 검색과 녹여내기 (Working Spec)

## 0. 이 문서의 위치
이번 작업 사이클의 working spec. 범위를 **aiLog 코어**로 좁힌다. 측정 해석 시스템 확장, 페르소나 파이프라인, UI 가시화 등은 별도 사이클에서 다룬다.

---

## 1. 이번 사이클의 목표

**자연어 발화에서 관련 episode를 찾아, 응답 안에 자연스럽게 녹여내는 것까지.**

성공 기준: 사용자가 *"이 시스템이 나를 기억하고 있구나"* 라고 명시적으로 느끼는 게 아니라, *"대화가 그냥 자연스럽게 이어진다"* 고 느끼는 상태. 메모리가 **드러나지 않으면서** 작동.

---

## 2. 비-목표 (이번 사이클에서는 하지 않는다)

- 페르소나 추출·갱신 파이프라인
- 멀티 사용자
- 측정 데이터 해석 도메인으로의 확장
- 페르소나/episode 가시화 UI
- 사용자가 episode를 직접 편집하는 기능
- raw 전문 검색

---

## 3. 데이터 구조

```
rawLog       대화 전체 원본 (불변, 진실의 원천)
   ↓
episode      rawLog 포인터 집합 + 의미텍스트
   ↓
의미텍스트    1인칭 회고 시점, episode 생성/갱신 시 만들어짐
              ── 검색 쿼리이자 검색 인덱스이자 녹여내기 재료
```

이 통일이 핵심이다. 시스템 전체가 *하나의 의미의 언어*로 작동한다.

---

## 4. 핵심 설계 원칙

### 4.1 1인칭 블로그 시점 고정
모든 의미텍스트는 *"~을 시도했는데 ~해서 ~로 바꿨다"* 같은 1인칭 회고로 작성. 외부 관찰자 시점(*"성훈이 ~을 했다"*) 금지.
이유: 톤 일관성 + 임베딩 응집도 + 녹여내기 자연스러움.

### 4.2 의미텍스트는 인과/흐름을 포함해야 함
사실 카드형 압축 금지. *왜 그랬는지, 그래서 어떻게 됐는지*가 함께 들어있어야 LLM의 어휘적 연결 작업 비용이 줄어 환각이 감소한다.

### 4.3 raw에 내려가지 않는 것이 기본
대부분의 retrieval은 의미텍스트로 충분히 작동해야 한다. 사용자가 *"그때 정확히 뭐라고 말했지"* 같은 디테일 검증을 명시적으로 요구할 때만 rawLog까지 내려간다.

### 4.4 보수적 소환
*꺼낼지 말지* 의 판단이 retrieval 품질의 절반. 모호하면 꺼내지 않는다. 어색한 소환이 누락보다 사용자 신뢰를 더 빠르게 깎는다.

---

## 5. 컴포넌트

### 5.1 의미텍스트 생성기 (episode 생성·갱신 시점)
- 입력: rawLog 묶음 (+ 기존 의미텍스트, 갱신 시)
- 출력: 1인칭 회고 시점 의미텍스트, 권장 길이 50~150 토큰
- 프롬프트는 외부 시점·사실 나열·메타 코멘트를 **명시적으로 금지**

### 5.2 검색기
- 의미텍스트의 OpenAI 임베딩 + 코사인 유사도 (기존 결정)
- 임계값은 보수적으로 (false positive보다 false negative 선호)
- 상위 N개 후보 반환 (N=3~5 출발)

### 5.3 후보 큐레이터 (메인 LLM 호출 전 단계)
검색기에서 받은 N개 후보 중 *현재 발화 맥락에 진짜 관련 있는 것만* 솎아낸다. 임베딩 유사도가 높아도 맥락이 다를 수 있다. 작은 LLM 호출 또는 룰 기반 필터.
**큐레이터는 생략 가능한 단계가 아니라 핵심 단계.** 빼면 LLM이 컨텍스트에 들어온 모든 episode를 어떻게든 쓰려고 한다.

### 5.4 녹여내기 (메인 LLM 호출)
큐레이터를 통과한 episode의 의미텍스트만 컨텍스트로 주입. 시스템 프롬프트에 명시:
- 메모리 출처를 언급하지 말 것 (*"기억하고 있어요"*, *"지난번에 말씀하신"* 류 금지)
- 인용이 아니라 *지금 문장에 녹이기*
- 관련 없다고 판단되면 *사용하지 않을 권리*가 있다
- 1턴에 한 episode만 활용. 과잉 소환 방지

### 5.5 관찰/평가 훅
어떤 발화에서 어떤 episode가 후보로 올라왔고, 큐레이터가 무엇을 떨궜고, 최종 응답이 무엇을 썼는지를 내부 로그로 보관. 자기 디버깅용, 사용자에게는 노출 안 함.

---

## 6. 구현 순서

1. 의미텍스트 생성 프롬프트 확정 → 기존 episode들에 일괄 재적용
2. 임베딩 인덱스 구축
3. 검색기 → 큐레이터 → 녹여내기 파이프라인을 직선으로 연결
4. 좋은 케이스 / 나쁜 케이스 각 10개의 회귀 테스트셋 구축
5. 시스템 프롬프트 반복 튜닝

---

## 7. 알려진 함정

- **의미텍스트 길이**: 너무 길면 임베딩 응집도가 깨지고 토큰 비용이 폭증, 너무 짧으면 사실 카드로 회귀해 녹여내기 품질 하락. 출발점 50~150 토큰, 실험으로 조정.
- **임계값**: 낮추면 어색한 소환이 폭증. 한 번의 어색한 응답이 신뢰를 크게 깎는다.
- **큐레이터 생략 유혹**: 파이프라인을 단순하게 만들고 싶어 큐레이터를 빼면 retrieval 품질이 무너진다.
- **시점 드리프트**: 의미텍스트 생성 프롬프트가 미묘하게 약해지면 외부 시점이 섞여 들어와 임베딩 공간 일관성을 무너뜨린다. 주기적으로 출력 샘플을 검사.

---

## 8. 검증

대화 종료 후 자기 평가 질문:
- 사용자가 메모리의 존재를 *눈치챘는가, 못 챘는가*. 못 챘으면 성공.
- 회상 부분이 *현재 문장에 녹아있는가, 따로 떠 있는가*.
- 비슷한 발화가 다시 들어왔을 때 같은 episode가 일관되게 소환되는가.
- 명백히 무관한 발화에서 episode가 *소환되지 않았는가* (false positive 통제).
